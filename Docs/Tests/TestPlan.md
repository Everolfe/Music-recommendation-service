## Введение

Данный документ описывает план тестирования приложения ["Менеджер музыкальной коллекции и рекомендаций"](https://github.com/Everolfe/Music-recommendation-service). 
Документ предназначен для людей, выполняющих тестирование данного проекта.
Цель тестирования - проверка соответствия реального поведения программы проекта и ее ожидаемого поведения, которое описано в [требованиях](https://github.com/Everolfe/Music-recommendation-service/blob/main/Docs/Requirements/SRS.md).

<a name="items"/>

## Объект тестирования

В качестве объектов тестирования можно выделить следующие функциональные требования:

* Вход пользователя в приложение;
* Регистрация нового пользователя;
* Поиск музыкальных композиций и информации о них;
* Управление музыкальной коллекцией;
* Получение персональных рекомендаций;
* Просмотр детальной информации о музыкальных произведениях;
* Выход пользователя из учётной записи.

<a name="quality"/>

## Атрибуты качества

1. Функциональность:
    - функциональная полнота: приложение должно выполнять все заявленные функции;
    - функциональная корректность: приложение должно выполнять все заявленные функции корректно.
2. Удобство использования:
    - интуитивно понятный интерфейс;
    - минималистичный дизайн;
    - адаптивность для различных разрешений экрана.
3. Производительность:
    - эффективная работа с локальной базой данных;
    - оптимальное использование API внешних сервисов.
    
<a name="risk"/>

## Риски

В данном случае к рискам можно отнести:
* Отсутствие интернет-соединения, что ограничит доступ к API Last.fm;
* Проблемы с доступностью API Last.fm;
* Ошибки при работе с локальной базой данных PostgreSQL;
* Некорректное отображение интерфейса на различных разрешениях экрана.

<a name="features"/>

## Аспекты тестирования

В ходе тестирования планируется проверить реализацию основных функций приложения. Аспекты, подвергаемые тестированию: 

### Авторизация и регистрация
Этот вариант использования небходимо протестировать на:
* Вход в приложение как гость;
* Вход под существующим пользователем;
* Выход из учетной записи;
* Валидация данных при регистрации.

### Управление профилем
Этот вариант использования небходимо протестировать на:
* Добавление музыкальных композиций в коллекцию; 
* Удаление композиций из коллекции;
* Отображение персональной коллекции; 

### Поиск и информация
Этот вариант использования небходимо протестировать на:
* Поиск музыкальных композиций через API;  
* Просмотр детальной информации о треках;  
* Поиск с различными параметрами (расширенный поиск).
 

### Рекомендации
Этот вариант использования небходимо протестировать на:
* Получение персональных рекомендаций;
* Обновление рекомендаций при изменении коллекции;
* Отображение рекомендаций для авторизованных пользователей.

### Интерфейс пользователя
Этот вариант использования небходимо протестировать на:
* Корректность отображения всех элементов интерфейса;
* Работоспособность навигации между экранами;
* Адаптивность под разные разрешения экрана.

### Функциональность для разных типов пользователей
Этот вариант использования небходимо протестировать на:
* Ограничения функционала для гостевых пользователей;
* Полный функционал для зарегистрированных пользователей;

## Нефункциональные требования:
* Производительность при работе с локальной базой данных;
* Безопасность хранения пользовательских данных;
* Удобство использования интерфейса.

## Выводы
Данный тестовый план позволяет протестировать основной функционал приложения "Менеджер музыкальной коллекции и рекомендаций". Успешное прохождение всех тестов не гарантирует полной работоспособности на всех версиях платформ и архитектурах, однако позволяет полагать, что данное программное обеспечение работает корректно. Особое внимание следует уделить тестированию интеграции с внешними API и работе приложения в условиях отсутствия интернет-соединения.
